
## 1. 전체 시퀀스 다이어그램

1. **구매자**: [구매하기] 버튼 클릭
2. **클라이언트(Frontend)**: 서버에 주문 생성 요청 (상품 ID, 수량 등)
3. **서버(Backend)**: 
    - 주문 데이터 생성 (상태: `READY`)
    - **중요**: 재고 미리 차감 (또는 선점)
    - 클라이언트에 `orderId`, `amount`, `orderName` 전달
4. **클라이언트**: 토스페이먼츠 SDK 호출 (`requestPayment`)
5. **토스페이먼츠**: 결제창 팝업/리다이렉트 -> 구매자가 카드 정보 입력 및 인증
6. **토스페이먼츠**: 인증 성공 시 `successUrl`로 리다이렉트 (쿼리 파라미터로 `paymentKey`, `orderId`, `amount` 전달)
7. **클라이언트**: 리다이렉트된 페이지에서 서버로 **'결제 승인 요청'** API 호출
8. **서버**: 
    - 전달받은 `amount`가 DB의 주문 금액과 일치하는지 검증
    - 토스페이먼츠 **Confirm API** 호출 (Secret Key 사용)
9. **토스페이먼츠**: 최종 결제 승인 결과 반환
10. **서버**: 주문 상태 변경 (`PAID`), 재고 확정, 구매 완료 처리
11. **클라이언트**: 결제 완료 화면 노출

---

## 2. 단계별 핵심 로직 (Backend 상세)

### Step 1: 주문 생성 및 재고 선점
구매자가 버튼을 누르자마자 서버는 주문 객체를 생성합니다. 이때 결제 도중 상품이 매진되는 것을 방지하기 위해 재고를 먼저 줄입니다.
- **Transaction 시작**
- `Product` 조회 (Pessimistic Lock 권장)
- `order` 테이블 Insert (상태: PENDING)
- `stock` 차감
- **Transaction 종료**

### Step 2: 결제창 호출 (Client-side)
토스 SDK에 서버에서 생성한 고유 `orderId`를 넘겨줍니다.


============

## 신경써서 관리해야 하는 부분

1.  **멱등성(Idempotency) 유지**: 사용자가 새로고침을 하거나 네트워크 오류로 인해 결제 승인 API가 두 번 호출될 수 있습니다. 동일한 `orderId`에 대해 중복 결제가 일어나지 않도록 처리가 필요합니다.
2.  **재고 복구 로직**: 결제창까지 띄웠으나 사용자가 중간에 이탈하거나, 최종 승인 단계에서 한도가 초과되어 실패할 경우 차감했던 재고를 다시 복구해주는 로직이 반드시 필요합니다.
3.  **검증 로직**: 클라이언트(JS)에서 넘겨주는 `amount`는 위변조가 가능합니다. 따라서 결제 승인 요청 시 반드시 DB에 저장된 주문 금액과 일치하는지 백엔드에서 대조해야 합니다.

내 기준 정리

1. 프론트에서 "구매하기" 버튼을 클릭하면 토스 페이먼트 sdk랑 연동해서 결제를 할 수 있는 화면을 보여준다. 
2. 구매하기 버튼을 누를때 백엔드 post/ order/ api에서 주문을 형성한다. 
3. 주문을 형성할때, 처음으로 들어온 수량을 **임시로** 저장해둔다.(나중에 비교할때 사용될 것)
    3-1. 주문을 생성할때 트랜잭션, 락 등 신경써서 만든다.
4. 주문이 만들어지고 orderId, amount, orderName(정확하지 않음)을 반환한다.
5. 프론트에서 orderId를 이용해서 성공 및 실패 URL(successUrl, failUrl)을 정의할 수 있다. 
6. 프론트에서 구매자, 카드사에 대한 정보를 입력하고 인증과정을 거친다. 
7. 인증이 완료되면, 구매자 입장에서는 모든 과정이 끝난것 처럼 보인다. 
8.  토스페이먼츠에서 결제 기록을 생성하고, 결제 키(paymentKey)를 발급(결제 키는 각 결제에 고유한 값이고, 결제 인증이 끝나면 자동으로 발급)
9. 토스페이먼츠에서 앱을 성공 URL로 리다이렉트한다.(https://docs.tosspayments.com/blog/what-is-successurl)(프론트에서 success.html을 만들어서 하는것이 일반적이다. 하지만 백엔드에서 api로 사용 가능)
예> 인증 성공 시, 미리 설정한 successUrl로 리다이렉트. <br>
URL 예시: .../success?paymentKey={key}&orderId={id}&amount={amount}
10. POST /orders/confirm (paymentKey, orderId, amount 전달) 등 결제 정보를 서버(백엔드)로 보내준다. 
11. 전달받은 orderId로 DB를 조회해 실제 결제 요청 금액(amount)이 일치하는지 확인 (위변조 방지).3번에 임시 저장해둔 값을 비교
12. 토스 승인 API 호출: Secret Key를 헤더에 넣고 **토스 서버**로 Confirm 요청 전송. 
13. 토스에서 최종 결제 처리 후 결과(성공/실패) 응답.
14. 결제 성공 시 주문 상태를 PAID로 변경.
15. 최종 응답을 프론트(클라이언트)에 전달.

> 의문점

- orderId의 형식이 있는걸까? 토스 페이먼츠 전용의 형식이 따로 있을까? 단순히 1,2,3 이런식으로 해도 될까?
    - 토스페이먼츠에서 orderId는 결제를 식별하는 고유한 값
    - 제약 사항: 영문 대소문자, 숫자, 특수문자(-, _)를 포함한 6자 이상 64자 이하의 문자열이어야 한다.

- paymentKey를 받았을 경우는 인증이 완료된 상태인데 여기서 만약 수량이 위조되거나 했으면 결제를 취소하고, 재고 수량을 롤백해야 하는가?
    - paymentKey를 받았을 때는 아직 돈이 빠져나가지 않은 상태.
    - 결제를 '취소'하는 게 아니라, 토스에 '승인 요청(Confirm)'을 아예 보내지 않으면 된다. 오류처리로 프론트에 요청을 보낸다.

- 토스에서 승인을 해준다고 했는데 그럼 위에서 검증 절차가 끝나고 단순히 헤더에 secertKey를 보내주는 것만으로 프론트에서 그걸 인식하고 결제가 승인된 것을 인지하는 것인가?
    - 백엔드가 토스에 Secret Key를 보낸다고 해서 프론트가 그걸 자동으로 아는 게 아니다. 
        - 백엔드 → 토스: "여기 시크릿키랑 티켓(paymentKey)이야. 결제 확정해!"
        - 토스 → 백엔드: "확인됐어. 여기 결제 완료 영수증 데이터야." (HTTP 200 OK)
        - 백엔드 → 프론트: "토스랑 확인 끝났어. 성공이야!" (결과를 JSON으로 응답)

- paymentKey가 결제하기 버튼을 누를 때 보내지는 건가?
    - 결제하기 버튼을 누를 때 보내는 게 아니라, 토스가 주는 것
    - 구매자가 카드 비밀번호를 입력하고 인증에 성공한 직후, 토스가 브라우저로 리다이렉트 시키면서 URL에 담아서 보내줍
    - 이 URL에서 키를 추출해 백엔드로 전달만 할 뿐

- 멱등적인 부분을 신경써야 하는데 만약 인증과정을 마치고 orderId를 백으로 보낸다고 했는데 이 시점에서 여러번 api가 호출되는 경우를 예방하면 되는건가?
    - orderId를 유니크 키로 활용

# 프론트와 백엔드의 역할 구분

> 프론트엔드 (Frontend) 의 역할
프론트엔드는 주로 "사용자와의 인터페이스" 및 **"토스 결제창 실행"**을 담당합니다.

- 결제 요청 준비: 사용자가 선택한 상품 정보를 백엔드에 보내서 orderId(주문번호)를 받아옵니다. 1번이 해당

- 토스 SDK 초기화: 토스에서 제공하는 JavaScript SDK를 로드합니다.

- 결제창 띄우기 (핵심): 백엔드에서 받은 orderId와 금액을 가지고 requestPayment() 함수를 실행합니다. 이때 결제창이 브라우저 위에 뜹니다.

- 리다이렉트 처리: 인증이 끝나면 토스가 보내주는 successUrl 혹은 failUrl 페이지에서 대기하다가, 성공 시 백엔드로 "승인해달라"고 신호를 보냅니다.

> 백엔드 (Backend) 의 역할
백엔드는 "돈과 재고의 무결성" 및 **"토스와의 실제 돈 거래"**를 담당합니다.

- 주문 데이터 생성: 프론트에서 요청이 오면 DB에 주문서를 쓰고, 재고를 차감한 뒤 고유한 orderId를 발급합니다.

- 결제 검증 (매우 중요): 프론트가 승인 요청을 보냈을 때, 프론트가 보낸 금액이 처음에 우리가 DB에 저장한 금액과 일치하는지 확인합니다. (사용자가 개발자 도구로 금액을 100원으로 고쳤을 수도 있으니까요!)

- 최종 승인 호출: 검증이 끝나면 백엔드 서버가 직접 토스 서버에 "인증 확인됐으니 진짜 결제 처리해!"라고 요청(Confirm API)을 보냅니다. 이때 API 비밀키를 사용합니다.

- 상태 업데이트: 결제가 완료되면 DB의 주문 상태를 결제 완료로 변경합니다.

# 궁금한 사항 정리
# secret_key 사용 방법

Secret Key는 응답(Response) 헤더가 아니라, 토스 승인 API를 호출할 때 요청(Request) 헤더에 들어간다.

- 누가 보내나: 우리 백엔드 서버

- 누가 받나: 토스페이먼츠 서버

## 어떻게 보내나: 
HTTP Basic Authentication 방식을 사용합니다. Secret Key 뒤에 콜론(:)을 붙여서 Base64로 인코딩한 값을 Authorization 헤더에 담습니다.

"Secret Key는 **절대 프론트엔드로 내려가면 안 됨**."

> 백엔드 승인(Confirm) 로직 흐름
프론트엔드에서 paymentKey, orderId, amount를 백엔드로 던져주면, 백엔드는 내부적으로 이런 일을 합니다.

1. 검증 (Validation)
토스에 요청하기 전에, 우리 DB를 먼저 확인합니다.

"우리 DB에 저장된 orderId의 결제 금액이 정말 amount와 같은가?"
<br>
"해당 상품의 수량이 충분한가?"
<br>
성공하면: 다음 단계인 토스 승인 요청으로 넘어갑니다.

2. 토스 승인 API 호출 (The "Confirm" Step)
여기서 Secret Key가 사용됩니다. 백엔드가 직접 토스 서버에 HTTP 요청을 보냅니다.

# 결제 요청을 할때 프론트에서 orderId(혹은 orderNumber)을 생성하면 안될까?

- 결제 위변조 위험: 악의적인 사용자가 프론트엔드 스크립트를 조작해 amount(금액)를 1,000,000원에서 100원으로 바꾸고, 본인이 만든 임의의 orderId를 토스로 보낼 수 있다. 백엔드는 이 주문이 진짜인지 가짜인지 알 수 없다.

- DB에 없는 주문의 탄생: 사용자가 결제창은 띄웠는데(프론트에서 orderId 생성), 결제 직전에 브라우저를 닫아버리면 어떻게 될까? 우리 백엔드 DB에는 그 주문 기록이 존재하지 않게 됩니다. 나중에 고객이 "돈은 나갔는데 주문 내역이 없어요!"라고 할 때 추적이 불가능.

- 중복 주문 번호: 클라이언트의 시간이나 로직에 의존해 번호를 만들면, 아주 희박한 확률이라도 중복된 orderId가 생성될 위험이 있습니다. 서버는 DB의 유니크 제약 조건을 통해 이를 완벽히 통제할 수 있지만, 클라이언트는 어렵다.

> ## 권장되는 정석 프로세스

1.  주문 생성 요청 (Client → Server): 사용자가 '결제하기'를 누르면 바로 결제창을 띄우지 않고, 먼저 우리 서버에 주문 정보를 보냅니다.

2. 가주문 저장 (Server): 서버는 DB에 주문 정보를 저장하고, **서버가 직접 생성한 고유한 orderNumber**를 생성합니다. 이때 주문 상태는 PAYMENT_PENDING (결제 대기) 입니다.

3. 결제 준비 응답 (Server → Client): 생성된 orderNumber와 서버가 계산한 **최종 금액(amount)**을 프론트로 내려줍니다.

4. 결제창 호출 (Client → Toss SDK): 프론트는 서버로부터 받은 검증된 값들로 토스 결제창을 띄웁니다.

# 추가로 생각할 부분

결제 취소(환불), 판매자에게 돈을 넣어주는 정산 부분, 결제 완료후 오류가 발생했을때(보상 트랜잭션, 웹 훅)

동시에 주문이 들어갈때 동시성, 결제 요청이 여러번 들어갈때의 멱등성
