4가지로 나뉜다.

## 동기/비동기 
![동기1](/Spring/Spring_images/동기1.png)

요청한 작업에 대해 완료여부를 신경써서 작업을 순차적으로 수행할지 아닌지에 대한 관점이다.

=>
- 동기 : 절차적이고 request의 맞는 순서에 따라 실행된다.

- 비동기 : 비절차적이고 requset에 순서와 관계없이 실행된다.

## 블록/논블록

현재 작업이 block(차단,대기)되느냐 아니냐에 따라 다른 작업을 수행할 수 있는지에 대한 관점이다.

=>
- 블록 : 작업을 중지시키는 것이다. a를 하다가 b가 들어오면 a를 중지키시고 b를 실행한 후 b가 완료되어야 a를 시킨다. 블록은 a에서 b로 제어권이라는 것이 넘어간다. 제어권이 이동한다

- 논블록 : 반대다. 작업을 중지시키지 않는다. a가 실행되면서 b를 실행시킨다. 여기서 a는 중지되지 않고 같이 실행된다. 제어권은 여전히 a에게 있다.


## 비동기 성능의 이점

![동기2](/Spring/Spring_images/동기2.png)

요청 작업의 완료여부를 신경쓰지 않고 자신의 그 다음 작업을 수행한다는 것은  I/O 작업과 같은 느린 작업이 발생할 때, 기다리지 않고 다른 작업을 처리하면서 동시에 처리하여 멀티 작업을 진행할수 있다는 것이다.
<br>
그렇게 되면 성능은 동기보다 좋아진다.

예를들어, 웹 애플리케이션에서 데이터베이스 쿼리를 수행하는 작업이 있다고 가정해보자. 이 작업을 만일 동기적으로 수행하면, 데이터베이스에서 응답이 올 때까지 기다려야 한다. 그러면 이 때 웹 애플리케이션은 다른 요청을 처리하지 못하므로, 대규모 트래픽이 발생할 경우 웹 애플리케이션의 성능이 저하될 수 있다. 하지만 비동기 방식으로 데이터베이스 쿼리를 수행하면, 데이터베이스에서 응답이 올 때까지 기다리는 동안에도 다른 요청을 '동시에 처리'할 수 있게 된다. 이렇게 비동기 방식을 사용하면, 대규모 트래픽에서도 안정적으로 동작할 수 있는 웹 애플리케이션을 만들 수 있다

    //서버 자원을 보호하기 위해 반드시 별도의 스레드 풀을 정의
    @Configuration
    @EnableAsync // 비동기 기능을 활성화합니다.
    public class AsyncConfig {

        @Bean(name = "taskExecutor")
        public Executor taskExecutor() {
            ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
            executor.setCorePoolSize(5);   // 기본 스레드 수
            executor.setMaxPoolSize(10);  // 최대 스레드 수
            executor.setQueueCapacity(500); // 대기 큐 크기
            executor.setThreadNamePrefix("AsyncThread-");
            executor.initialize();
            return executor;
        }
    }

- 서비스 계층

    //@Async를 붙이면 이 메서드는 메인 스레드가 아닌 위에서 만든 taskExecutor 스레드에서 돌아간다.

    @Service
    @Slf4j
    public class RoutineService {
        
        // 1. 외부 건강 데이터 서버에서 데이터를 가져오는 비동기 I/O 작업
        @Async("taskExecutor")
        public CompletableFuture<String> fetchUserHealthData(Long userId) {
            log.info("외부 데이터 조회 시작: {}", userId);
            
            try {
                // 실제로는 RestTemplate이나 WebClient가 들어가는 자리입니다.
                Thread.sleep(2000); 
            } catch (InterruptedException e) {
                e.printStackTrace();
            }
            
            return CompletableFuture.completedFuture("걸음 수: 10,000보 (성공)");
        }

        // 2. 루틴 이행 여부를 분석하는 CPU 집약적 작업
        public String analyzeRoutine(String data) {
            log.info("데이터 분석 중...");
            return data + " -> 분석 결과: '매우 우수'";
        }
    }

----

- 컨트롤러

    @RestController
    @RequiredArgsConstructor
    public class RoutineController {

        private final RoutineService routineService;

        @GetMapping("/api/routine/{userId}")
        public CompletableFuture<ResponseEntity<?>> getRoutineReport(@PathVariable Long userId) {
            log.info("루틴 리포트 요청 수신");

            // 1. 비동기로 데이터 조회를 던져놓음 (Blocking 되지 않음)
            return routineService.fetchUserHealthData(userId)
                    .thenApply(data -> {
                        // 2. 조회가 완료되면 이어서 분석 진행
                        String analysis = routineService.analyzeRoutine(data);
                        return ResponseEntity.ok(analysis);
                    })
                    .exceptionally(ex -> ResponseEntity.status(500).body("오류 발생"));
        }
    }

### 코드의 실행순서

1. 컨트롤러 호출 : 사용자가 API를 호출함
2. 비동기 메서드 실행 : fetchUserHealthData()가 호출. 이때 내부적으로는 **별도의 스레드(AsyncThread-1)**가 생성되어 2초간 데이터를 가져오기 시작한다.
3. 예약(Registration): **컨트롤러는 기다리지 않고 바로 다음 줄인 .thenApply(...)를 만난다**. 이때 "데이터가 다 오면 이 analyzeRoutine을 실행해줘"라고 포스트잇(예약권)을 붙여놓고, 컨트롤러는 자기 일을 끝냅니다. **(여기서 컨트롤러 스레드는 자유의 몸이 됩니다.)**
4. 2초 동안 AsyncThread-1은 데이터를 가져옴.
5. 실행(Trigger): 2초 뒤 데이터 조회가 완료되는 순간, 아까 붙여둔 **포스트잇(thenApply)**이 자동으로 실행.

--------

### 이해를 돕기 위한 코드


    @GetMapping("/api/test")
    public CompletableFuture<String> asyncTest() {
        log.info("### [Step 1] 컨트롤러 진입 (스레드: {})", Thread.currentThread().getName());

        CompletableFuture<String> future = service.fetchData() // 비동기 작업 시작
            .thenApply(data -> {
                log.info("### [Step 3] 예약된 작업 실행 (스레드: {})", Thread.currentThread().getName());
                return data + " 완료";
            });

        log.info("### [Step 2] 컨트롤러 나감 (스레드: {})", Thread.currentThread().getName());
        return future;
    }

> 비동기 로그 찍히는 순서

1. step1 
2. step2
3. step3

> 동기적일 경우 로그 찍히는 순서

1. step1
2. step3 (2초간 대기해야한다.)
3. step2

=> 이렇게 동기적일 경우 2초동안 묶여서 아무것도 할 수없다. 하지만 비동기의 경우 step2를 완료시킨 후에 자유의 몸이 된다. 그 이후에 하는 일은 다**른 사용자의 HTTP요청을 처리하는 것**이다.

## 비동기와 논블로킹의 구분

둘은 시점과 관련된 이론적 개념이라 실제 코드에서 경계를 구분하기 애매하다. 그래서 프로그래밍에서 이를 혼용해서 사용한다. 하지만 엄연히 차이가 있다.

**비동기는 출력 순서와 관련된 개념이고 논블로킹은 병렬 실행과 관련된 개념이다.**


# 동기/비동기 + 블로킹/ 논 블로킹 조합

![동기3](/Spring/Spring_images/동기3.png)

총 4가지로 만들어질 수 있고 프로그램이 작동하는 방식에 영향을 미친다.

### 동기 + 블록


다른 작업 결과가 자신의 작업에 영향을 주는 경우에 활용할 수 있다. 

1. 팀장 : 사원1씨 업무 A 좀 해주세요
2. 사원1 : 네 알겠습니다. (A를 처리중)
3. 팀장 : (사원1이 A를 다할때까지 아무일도 하지 않고 기다린다)
4. 사원1 : 팀장님 A 업무 완료했습니다.
5. 팀장 : 수고했어요. 사원2씨 업무 B좀 해주세요.
6. 사원2 : 네 알겠습니다. (B를 처리중)
7. 팀장 : (사원2가 B를 다할때까지 아무일도 하지 않고 기다린다)
8. 사원2 : 팀장님 B 업무 완료했습니다.
9. 팀장 : 수고했어요. 사원3씨 업무 C좀 해주세요.
...생략


=>
request를 받은 순서로 실행되고 한 작업이 끝나야 다음 작업을 할 수 있다.

------

### 동기 + 논블럭 


다른 작업이 진행되는 동안에도 자신의 작업을 처리하고 다른 작업의 결과를 바로 처리해 작업을 순차대로 수행하는 방식이다. 

1. 팀장 : 사원1씨 업무 A 좀 해주세요
2. 사원1 : 네 알겠습니다. (A를 처리중)
3. 팀장 : 다음 업무 B를 하려면 A가 완료되야 하는데.. 사원1씨 다했어요?
4. 사원1 : 아직이요 A 처리중입니다
5. 팀장 : 사원1씨 다했어요?
6. 사원1 : 아직이요 A 처리중입니다
7. 사원1 : 팀장님 A 모두 완료했습니다
8. 팀장 : 수고했어요. 사원2씨 업무 B좀 해주세요.
9. 사원2 : 네 알겠습니다. (B를 처리중)
10. 팀장 : 다음 업무 C를 하려면 B가 완료되야 하는데.. 사원2씨 다했어요?

=> request를 받은 순서로 실행은 된다. 하지만 작업을 멈추지 않는다. a를 실행하면서 b가 실행이 완료되었는지 계속 확인한다. b가 완료되었다면 c를 실행시킨다. 순차적으로 진행되지만 a는 작업을 멈추지 않을 수 있다.
논블록으로 인해 계속 실행되지만 각 태스크는 서로 영향을 주는 관계에 있다.

--------

### 비동기 + 블럭 

**안티 패턴이다.** 실무에서 잘 나오는 경우가 없고 실수 할 때 발생하는 경우이다.

=> request의 순서에 상관없이 실행되어야 하지만 블럭으로 인해 동기+블럭 처럼 실행된다.

------

### 비동기+논블럭

다른 작업이 진행되는 동안에도 자신의 작업을 처리하고 (Non Blocking), 다른 작업의 결과를 바로 처리하지 않아 작업 순서가 지켜지지 않는 (Async) 방식

1. 팀장 : 사원1씨 A업무좀 해주세요. (동시에 지시)
2. 팀장 : 사원2씨 B업무좀 해주세요. (동시에 지시)
3. 팀장 : 사원3씨 C업무좀 해주세요. (동시에 지시)
4. 팀장 : 다른일을 해야지 ~
5. 사원2 : 팀장인 B 모두 처리했습니다. (업무량에 따라 각 사원마다 완료하는 시간이 제각기 다를 수 있다)
6. 사원1 : 팀장인 A 모두 처리했습니다.
7. 사원3 : 팀장인 C 모두 처리했습니다.



=> request의 순서에 상관없이 a가 실행되고 b가 실행된다. 작업은 당연히 중단되지 않는다. b가 좀 대기가 필요하더라도 신경쓰지 않고 a,c는 작업을 이어간다. b의 대기가 끝나면 그때 b를 실행한다. 
각 작업이 서로 영향을 주지 않는다.
