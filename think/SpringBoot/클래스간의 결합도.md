
### 기존 코드

    @RequiredArgsConstructor
    @Service
    public class UserService {

        private final UserRepository userRepository;
        private final PasswordEncoder passwordEncoder;

        public SiteUser create(String username, String email, String password) {
            SiteUser user = new SiteUser();
            user.setUsername(username);
            user.setEmail(email);
            user.setPassword(passwordEncoder.encode(password));
            this.userRepository.save(user);
            return user;
        }
    }


이곳에서 문제라기 보다는 저런식으로 함수내부에 new을 사용해서 만들면 서비스와 엔티티간의 의존도가 증가한다고 생각했다. 
그래서 추상화시켜야겠다고 생각했다. 

인터페이스를 만들고 클래스 하나를 더 만들어서 implements를 받았다. 
---
    public interface Creatable_user {
        SiteUser create_new();
    }

    @Component
    public class CreateUser implements Creatable_user{

        @Override
        public SiteUser create_new() {
            SiteUser user = new SiteUser();
            return user;
        }

    }


위의 user을 return으로 받았다.

    private final CreateUser createUser;
    ...
    SiteUser user = createUser.create_new();

실행되는 것에는 문제가 없었지만 만들고 보니 오히려 코드 이해가 복잡해진 것 같았다. 

Ai에게 물어보니 over-engineering이 맞다고 한다. 

불필요한 복잡성을 추가했고 도메인 지식이 파편화되었다고 한다. 유저를 생성하는 로직은 유저 도메인이 핵심인데 이것을 외부 클래스로 빼버림으로써 가독성이 떨어졌다는 것이다. 
또한 생성 방식의 제한이 있었다. 만약 생성 시점에서 파라미터가 바뀌면 인터페이스 구조자체가 흔들릴 수 있다. 
=> 그렇네, 만약 닉네임이라는 새로운 파라미터가 들어온다면 인터페이스에서 선언한 메서드가 수정되야 한다. 의존성을 줄이려 했는데 반대로 더 의존적인 상황이 만들어짐.



# AI가 준 위의 문제 해결 방안

    // SiteUser 엔티티 내부
    public static SiteUser create(String username, String email, String password) {
        SiteUser user = new SiteUser();
        user.setUsername(username);
        user.setEmail(email);
        // password 암호화는 서비스나 별도 도메인 서비스에서 처리 권장
        return user;
    }

엔티티 내부에 선언하는 메서드를 만들어서 서비스 레이어의 의존성을 줄일 수 있다. 혹은 @Builder을 사용하는 방식도 있다.
