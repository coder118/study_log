# 프로세스

프로그램(코드 덩어리)를 실행시켜서 정적인 프로그램을 동적으로 변해 **프로그램이 돌아가고 있는 상태**

<br>

프로그램을 실행하기 위해서는 OS가 시스템 자원(CPU)을 할당 받아 프로그램 코드를 실행시켜야 한다.


# 스레드

과거에는 하나의 프로세스만 사용해도 괜찮았지만 기술이 발전되면서 프로세스 한개로 프로그램을 실행하는 것에 한계가 생겼다.

<br>

스레드란, 하나의 프로세스 내에서 동시에 진행되는 작업 갈래, 흐름의 단위를 말한다.
<br>

[동기와 비동기 참고하면 도움이된다.](/Spring/동기와%20비동기&블로킹과%20논블로킹.md)

이해하기 쉽게 설명해보자면 크롬 웹 브라우저 상단 탭 하나하나가 프로세스라고 생각하고 스레드를 웹 브라우저 단일 탭 내부의 브라우저 이벤트 루프, 네트워크 처리, I/O 및 기타 작업을 관리하고 처리하는 것이라 보면된다.(이벤트 루프에 하나의 스레드가 있는 것. 위의 예시의 경우는 멀티 스레드로 이해하면 된다.)


# 프로세스와 스레드의 메모리 구조

> 프로세스 실행 시점 메모리 영역

![프로세스1](/CS/OS/OS-imgaes/프로세스1.png)


- 코드 영역(Code / Text) : 프로그래머가 작성한 프로그램 함수들의 코드가 CPU가 해석 가능한 기계어 형태로 저장되어 있다.
- 데이터 영역(Data) : 코드가 실행되면서 사용하는 전역 변수나 각종 데이터들이 모여있다. 데이터영역은 .data ,.rodata, .bss 영역으로 세분화 된다.

    - .data : 전역 변수 또는 static 변수 등 프로그램이 사용하는 데이터를 저장
    - .BSS : 초기값 없는 전역 변수, static 변수가 저장
    - .rodata : const같은 상수 키워드 선언 된 변수나 문자열 상수가 저장

- **스택 영역(Stack)** : 지역 변수와 같은 임시적인 자료를 저장하는 독립적인 공간이다. Stack은 함수의 호출과 함께 할당되며, 함수의 호출이 완료되면 소멸한다. **동적 영역**
- **힙 영역(Heap)** : 생성자, 인스턴스와 같은 동적으로 할당되는 데이터들 저장공간.사용자에 의해 메모리 공간이 동적으로 할당되고 해제된다. **동적영역**


> 스레드의 자원 공유


![프로세스2](/CS/OS/OS-imgaes/프로세스2.png)

![프로세스3](/CS/OS/OS-imgaes/프로세스3.png)

프로세스 내에서 스레드는 Stack만 할당 받고 복사되어서 code,data,heap의 내용을 공유받는다.
즉, 프로세스 내 여러개의 스레드가 내용을 공유하면서 사용한다는 것이다.
<br>
생각해볼 포인트 => 왜 stack만 독립적으로 스레드가 가질까? 독립적 stack을 가진다는 소리는 독립적 함수 호출이 가능해진다. 그말은 스레드가 독립적인 실행 흐름을 가질 수 있다는 소리.


### 프로세스의 자원 공유

프로세스 별로 별도의 메모리를 사용하므로 한 프로세스는 다른 프로세스의 변수나 자료구조에 접근이 불가능한데 어떻게 가능한 걸까?

1. IPC(inter-process communication)사용
2. LPC(local inter- process communication) 사용
3. 별도로 공유 메모리를 만들어서 정보를 주고 받도록 설정

# 프로세스와 스레드의 동시 실행 원리


## 병렬성

직관적으로 명령어를 메모리에서 뽑아 해석하고 실행하는 반도체 유닛인 여러개의 코어에 맞춰 여러개의 프로세스, 스레드를 돌려 병렬로 작업들을 동시 수행하는 것을 말한다.


## 동시성

병렬성과 유사해보이지만 좀 다르다. Parallelism가 물리적으로 **정말로 동시에 실행하는 것**이라고 하면, Concurrency는 **동시에 실행하는 것처럼 보이게 하는 것**으로 이해하면 된다.

![프로세스4](/CS/OS/OS-imgaes/프로세스4.png)

단, 이때 작업들을 번갈아가면서 실행할때 작업들을 아주 잘게 나누어 아주 조금씩만 작업을 수행하고 다음 작업으로 넘어가는 식으로 동작
이렇게 진행 중인 작업들을 A → B → C → D 로 번갈아 바꾸는 것을 Context Switching 이라고 부른다.

---

### 병렬성 쓰면 되는데 왜 동시성이 필요할까?

1. 하드웨어적 한계: CPU의 발열과 넣을 수 있는 코어 수의 한계가 있다.
2. 논리적인 효율 : 총 4개의 작업을 처리할 수있는 프로세스가 있는데 2개가 먼저 가벼운 작업을 수행할때는 남은 2개가 다른 작업을 할 수 있다. 하지만 4개가 전부 무거운 작업을 하게된다면? 끝날때까지 다른 일을 하지 못한다. 이를 극복하기 위해서 작업을 아주 잘게 나눠 번갈아가면서 처리하는 것이다.

적절히 병렬성과 동시성을 섞어가며 활용해줘야 한다. 


# 프로세스 컨텍스트 스위칭

이걸 알기 위해서 프로세스 스케줄링, 상태, 상태전이,PCB에 대해 알아야 한다. 간단히 살펴보자.

## 프로세스 스케줄링

운영체제에서 CPU를 사용할 수 있는 프로세스를 선택하고, CPU를 할당하는 작업을 말한다.

1. FCFS
2. SJF
3. Priority
4. RR(Round-robin)

## 프로세스 상태 + 상태 전이

![프로세스5](/CS/OS/OS-imgaes/프로세스5.png)


프로세스 상태는 프로세스가 실행되는 동안 변경되는 고유 상태를 의미하고 상태 전이는 프로세스가 실행되는 동안 OS에 의해 변경되는 것을 말한다.

> 상태

- 생성 (new): 프로세스가 생성되고 아직 준비가 되지 않은 상태
- 준비(ready) :CPU를 할당받을 수 있는 상태이며, 언제든지 실행될 준비가 되어있다.
- 실행 : 프로세스가 CPU를 할당받아 실행되는 상태
- 대기 : 프로세스가 특정 이벤트(입출력 요청 등)가 발생하여 대기하는 상태,CPU를 할당받지 못함.
- 종료 : 프로세스가 실행을 완료하고 종료된 상태, 메모리에서 제거

> 상태 전이

- Admitted (new → ready) : 프로세스 생성을 승인 받음(프로그램 동작)
- Dispatch (ready→ running) : 준비 상태에 있는 여러 프로세스들 중 하나가 스케줄러에 의해 실행됨
- Interrupt (running → ready) : Timeout, **예기치 않은 이벤트가 발생**하여 현재 실행 중인 프로세스를 준비 상태로 전환하고, 해당 작업을 먼저 처리
- I/O or event wait (running → wainting) : 실행 중인 프로세스가 **입출력이나 이벤트를 처리**해야 하는 경우, 입출력이나 이벤트가 끝날 때까지 대기 상태로 전환
- I/O or event completion (waiting → ready) : 입출력이나 이벤트가 모두 끝난 프로세스를 다시 준비 상태로 만들어 스케줄러에 의해 선택될 수 있는 상태로 전환


## PCB(process control block)

프로세스 스케줄링을 위해 프로세스에 관한 모든 정보 저장하는 임시 저장소

> 하는 일

프로세스를 관리하며, 프로세스의 실행 상태를 파악하고, 우선순위를 조정하며, 스케줄링을 수행하고, 다른 프로세스와의 동기화를 제어


## 컨텍스트 스위칭

위의 상태전이가 빠르게 동작된다고 생각하면된다.
<br>
여러 개의 프로세스를 번갈아가며 실행하여 CPU 활용률을 높이기 위해 컨텍스트 스위칭이 필요한 것
<br>
동작 중인 프로세스가 대기를 하면서 해당 프로세스의 상태(Context)를 보관하고, 대기하고 있던 다음 순서의 프로세스가 동작하면서 이전에 보관했던 프로세스의 상태를 복구하는 작업

# 스레드 컨텍스트 스위칭

프로세스 컨텍스트 스위칭과 유사하게 스레드 스케줄링, 상태, 상태 전이, TCB에 관해 알아야 한다.

## 스레드 스케줄링

CPU를 사용할 수 있는 스레드를 선택하고, CPU를 할당하는 작업

- Round Robin 
- Priority-based scheduling
- Multi-level Queue scheduling

## 스레드 상태와 상태 전이


![프로세스8](/CS/OS/OS-imgaes/프로세스8.png)

- NEW : 스레드가 생성되고 아직 호출되지 않은 상태

- RUNNABLE :스레드가 실행되기 위해 기다리는 상태CPU를 할당받을 수 있는 상태.

- BLOCKED : 스레드가 특정 이벤트(입출력 요청 등)가 발생하여 대기하는 상태. **CPU를 할당받지 못하며,** 이벤트가 발생하여 다시 RUNNABLE 상태로 전환될 때까지 대기한다.

- TERMINATED : 스레드가 실행을 완료하고 종료된 상태더 이상 실행될 수 없으며, 메모리에서 제거된다.


## TCB(thread control block)

스레드의 상태 정보, 스레드 ID, 스레드 우선순위, 스케줄링 정보 등 다양한 정보를 저장한다.TCB도 스레드가 생성될 때 운영 체제에 의해 생성된다.
- 스레드 간의 자원 공유와 동기화도 TCB를 사용하여 관리
    - 예를 들어, 뮤텍스(mutual exclusion)나 세마포어(semaphore)와 같은 동기화 기법을 사용할 때, TCB에서 해당 스레드의 뮤텍스나 세마포어 정보를 관리하고, 스레드가 해당 자원에 대한 접근 권한을 획득하거나 반납할 때 TCB의 정보를 업데이트하게 된다.

> 임계 영역

멀티 스레딩 환경에서 임계영역은 공유자원을 접근하는 코드 영역을 말함. 전역변수, heap 메모리 영역

> 뮤텍스(Mutex)

상호 배제 기법 중 하나, 임계영역에 들어가기 전에 락을 획득하고, 임계 영역을 나올떄 락을 해제하여 다른 스레드가 접근 할 수 있도록 한다. 오직 한개의 스레드만 공유 자원에 접근을 허용

> 세마포어

동시에 접근 가능한 스레드의 개수를 지정할 수 있다. 1개로 지정하면 뮤텍스와 동일한 기능. 2 이상이면 동시에 접근 가능한 스레드 수를 제어할 수 있다.

#### 데드락=교착상태(멀티 스레드, 프로세스 환경의 문제점)

여러 개의 스레드가 서로 대기하며 무한정 기다리게 되는 무한 루프와 같은 증상을 말한다.

![프로세스9](/CS/OS/OS-imgaes/프로세스9.png)

이런 현상은 상호배제(Mutual Exclusion), 점유와 대기(Hold and Wait), 비선점(No Preemption), 순환 대기(Circular Wait) 등의 알고리즘을 통해 극복해야 한다.
