# 트랜잭션이란?

데이터베이스 상태를 변경시킬때 수행하는 작업단위이다. 한꺼번에 수행되어야 할 연산들을 말한다. 

- update, insert, delete 같은 연산이 포함된다.

트랜잭션은 데이터의 일관성을 보장하는 역할을 한다. 만약 중간에 작업이 성공적으로 완료되지 않으면 rollback을 사용해 변경사항이 없었던 상태로 돌려서 무결성을 유지한다.
만약 작업이 성공적이라면 commit해서 변경상태를 저장한다.

## ACID

- 원자성(atomicity) : all or nothing
- 일관성(consistency) : 모든 작업을 성공적으로 완료된 후, DB 상태가 일관성을 유지해야 한다.
- 격리성(isolation) : 각각의 트랜젹션은 서로 영향을 주지 않아야 한다. 락과 같은 잠금 메커니즘.
- 지속성(durability) :  트랜잭션이 완료되었을때, 결과는 영구적으로 반영되어야 한다. 시스템이 문제를 일으켜도 트랜잭션 결과는 손실되지 않아야 한다.


## 트랜잭션을 하나씩 진행하면 문제가 없다. 하지만 효율성이 떨어진다. 그럼 어떻게 해야 ACID를 지킬 수 있을까?

격리성 수준을 조정해 여러 트랜잭션을 동시에 처리할 수 있게 해야 한다.

# 동시성과 일관성

트랜잭션의 일관성이 완전히 보장될 경우, 다수의 request가 들어왔을때 응답의 지연이 발생되는 현상이 발생할  수 있다. 트랜잭션 하나가 실행되면 락이 걸리고 다른 트랜잭션은 대기상태에 빠지게 되기 때문이다. 
<br>
위 문제를 해결하기 위해 트랜잭션이 동시에 데이터에 접근하게 하면 데이터의 일관성을 해칠 수 있다.

<br>

이처럼 동시성과 일관성은 트레이드 오프 관계에 있기 때문에, 둘 사이의 적절한 균형이 필요하다.



![트랜잭션1](/CS/DB/DB_images/트랜잭션1.png)


균형을 이루기 위해 **동시성 제어**가 필요한다. 이는, 동시에 실행되는 트랜잭션의 수를 최대화하며 동시에 데이터 일관성을 보장하는 역할이다. 

1. 낙관적 동시성 제어(Optimisitc) : 같은 데이터를 동시에 수정하지 않을 것으로 가정한다.(낙관적) 이는 트랜잭션이 데이터를 읽는 시점에 락을 설정하지 않는다는 말이다. 대신 수정하려 할때,데이터가 자신이 읽은 이후에 변경되었는지 확인한다. 
    - 이런 방법은 데이터 충돌이 드문 경우 효과적이고, 대기시간을 줄일 수 있다.

2. 비관적 동시성 제어(Pessimistic) : 반대다. 같은 데이터를 동시에 수정할 것으로 가정(비관적). 이는 트랜잭션이 특정 데이터에 대한 작업을 시작하려할때 락을 설정한다는 것이다. 그래서 락을 해제할때까지 다른 트랜잭션은 접근할 수 없다. 
    - 일관성 높이는 것에 도움이 되지만, 동시에 여러 트랜잭션이 동일한 데이터 접근하려 할 경우 대기시간이 길어진다.

낙관적, 비관적 제어는 **락을 언제 어떻게 사용할지 정하는 전략이라고 보면된다**.

# 동시성 제어의 문제점들

1. Dirty Read
2. Non Repeatable Read
3. Phantom Read

## Dirty Read

**한 트랜잭션이 아직 커밋되지 않은 다른 트랜잭션의 변경을 읽는 경우.** 이때 해당 트랜잭션이 Rollback될 경우 최종 결과값이 비 일관적으로 적용될 가능성이 있다.

예시>

1. A트랜잭션에서 회원 a의 구매 목록을 추가해 결제금액이 4만원 -> 5만원으로 변경
2. A가 아직 커밋되지 않았는데 B트랜잭션이 a의 결제 금액을 조회
3. 결제금액이 5만원으로 조회된다.(아직 커밋되지 않은 상황, Dirty Read)
4. A 트랜잭션이 rollback되며 5만원->4만원으로 변경
5. 하지만, B 트랜잭션은 이전에 읽은 5만원으로 결제를 진행

<br>
이처럼 Dirty Read 문제가 발생하면 데이터의 일관성이 깨질 수 있다.

## Non Repeatable Read

**한 트랜잭션 내에서 같은 쿼리를 두번 수행했을때, 두 쿼리의 결과가 서로 다른 경우를 말함.**

주로, 데이터를 update하거나 delete할때 발생함. 

예시>

1. B 트랜잭션이 회원a의 결제금액을 조회
2. 4만원 조회 (1번째 조회)
3. A 트랜잭션이 회원 a의 결제금액을 5만원으로 변경하고 commit 
4. B트랜잭션이 회원 a의 결제 금액을 다시 조회
5. 5만원이 조회된다.(2번째 조회)

2,5에서 같은 쿼리를 실행했는데 다른 결과가 나온 것을 확인할 수 있다.

## Phantom Read

한 트랜잭션 내에서 **동일한 쿼리를 두 번 이상 실행**할 때, 첫 번째 쿼리에서는 없던 레코드가 **두 번째 쿼리에서 나타나거나, 반대로 사라지는 현상**

예시>

1. 트랜잭션 A가 결제금액이 4만원 이상인 회원을 검색-> 회원 a와 회원 b가 조회
2. 트랜잭션 B에서 회원 c의 결제 금액을 5만원으로 변경하고 commit
3. 트랜잭션 A에서 결제금 4만원이상 다시 조회-> 회원 a,b,c가 조회된다.(c라는 회원이 생김)

원래 없던 회원 c가 생겨난 경우이다. 
<br>
데이터를 읽은 후 로직을 진행하는데 있어서, 다른 트랜잭션의 Update, Insert의 개입으로 다시 읽었을 때 데이터의 무결성이 깨진다.

# 락

- 특정 작업을 수행하는 동안 다른 트랜잭션의 접근으로부터 보호하는 방법이다. 
- 데이터의 일관성을 보장하고, 동시에 수행되는 트랜잭션간의 충돌 방지를 위해 사용된다. 

> 락이 보통 해제되는 시점

트랜잭션이 commit되거나, rollback될 경우

## 락의 종류

1. 공유 락(shared lock) : 여러 트랜잭션이 **동시에 같은 데이터를 읽게** 해준다. 단, 그 데이터의 **쓰기 작업은 허용하지 않는다**. 
    - 한 트랜잭션이 읽는 동안 다른 트랜잭션은 해당 데이터를 변경할 수 없다.
2. 배타적 락(Exclusive lock) : **한 개의 트랜잭션**만 데이터를 **읽고 쓰게** 해준다. 해당 트랜잭션의 락이 해제될때까지 **다른 모든 트랜잭션 접근을 차단**한다.
    - 한개의 트랜잭션만 데이터에 접근이 가능하고 나머지 요청은 대기 상태가 된다.

# Isolated level(격리 수준)

동시에 트랜잭션이 처리될때, 특정 트랜잭션(a)이 다른 트랜잭션(b)에서 변경하거나 조회하는 데이터를 볼 수 있게 허용할지 말지를 결정하는 것.

- READ UNCOMMITTED(가장 낮은 격리 수준)
- READ COMMITTED (커밋된 데이터만 읽을 수 있는 격리 상태)
- REPEATABLE READ(반복 읽기)
- SERIALIZABLE

## READ UNCOMMITTED(가장 낮은 격리 수준)

![트랜잭션2](/CS/DB/DB_images/트랜잭션2.png)

각 트랜잭션에서의 변경 내용이 COMMIT이나 ROLLBACK 여부와 상관 없이 다른 트랜잭션에서 값을 읽을 수 있는 격리 수준이다. => update,insert로 변하고 있는 상태를 select같은것으로 조회 가능

> 생길 수 있는 문제점

Dirty Read, Non-Repeatable Read, Phantom Read가 다 발생할 수 있다.

## READ COMMITTED

커밋된 데이터만 읽을 수 있는 격리 상태, 대부분 RDB에서 기본적으로 사용.

이때 실제 테이블에서 가져오는 것이 아니라 Undo 영역에 백업된 레코드에서 값을 가져온다.
=> 트랜잭션이 중단되었을때 원래 값으로 돌리기 위해서 Undo영역에 임시로 값을 저장해두고 rollback할때 해당 값을 사용함.

![트랜잭션3](/CS/DB/DB_images/트랜잭션3.png)


> 생길 수 있는 문제

위 그림을 Jeju로 바뀌었지만 아직 커밋되지 않은 상태라 트랜잭션2는 확인할 수 없다. 커밋 후에 Jeju를 확인할 수 있다. 이렇게 되면 Dirty Read 문제가 발생하지 않지만, REPEATABLE READ의 정합성에 어긋나는 경우가 발생할 수 있다. 
<br>
주로, 입금, 출금 처리가 진행되는 금전적인 처리에서 주로 발생함.

![트랜잭션4](/CS/DB/DB_images/트랜잭션4.png)


## REPEATABLE READ(반복 읽기)

**한 트랜잭션에서 내에서 같은 쿼리를 여러번 실행할 경우, 언제나 동일한 결과를 가져와야 함**(위의 사례를 확인하자, 똑같은 쿼리를 호출했지만 한번은 busan 한번은 jeju가 나옴)

<br>

이렇게 하기 위해, 트랜잭션마다 고유한 ID를 부여하고, 해당 ID보다 작은 트랜잭션 번호에서 변경한 데이터만 읽게 된다.

<br>

이는 일종의 MVCC(Multi Version Concurrency Control)방기법으로, 하나의 데이터에 대해 **여러 버전을 유지**함으로써 **동시성을 향상시키는 동시에, 일관성을 보장**합니다.

> MySQL의 Undo Log

MVCC를 구현하기 위한 것. 데이터의 이전 버전을 저장하는 공간이다. **트랜잭션 도중에** 데이터의 변경이 필요할 경우 이 공간에서 **이전 데이터를 가져온다**.

<br>
주의점은 Undo Log가 쌓이면 MySQL서버 처리 성능이 떨어질 수 있으므로, 주기적 삭제가 필요함.

![트랜잭션5](/CS/DB/DB_images/트랜잭션5.png)

> 발생할 수 있는 문제

Dirty Read, Non Repeatable Read는 발생하지 않지만, Phantom Read 문제가 발생할 수 있다.

![트랜잭션6](/CS/DB/DB_images/트랜잭션6.png)

새로운 값을 insert를 할 경우 2로 받아야 하는데 3이라는 값을 받게 되므로, 이런 현상을 방지하기 위해 **쓰기 잠금을 걸어야 한다**.
<br>
update의 경우는 변경 전 사항을 Undo Log에 저장하면 되지만 insert의 경우는 변경 전 테이블을 저장해 둘 수 없어서 생기는 문제다.

## SERIALIZABLE(가장 높은 격리 레벨)

어떠한 동시성 문제도 발생하지 않지만, 성능 측면에서 동시성이 가장 떨어진다.
