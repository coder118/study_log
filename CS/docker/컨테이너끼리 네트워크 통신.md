
# 컨테이너 통신 방법

기본적으로 모든 도커 컨테이너는 동일한 브릿지 네트워크에 연결된다. 여기서, 같은 브릿지 네트워크에 속한 컨테이너들은 서로 자유롭게 통신할 수 있다. 

- 각 컨테이너는 고유한 IP 주소를 할당받습니다.

1. IP 주소로 직접 통신하는 방법 -> 같은 네트워크에 있어야 한다.

2. 컨테이너 이름으로 통신 -> 같은 네트워크 + 사용자 정의 네트워크

3. host를 거쳐서 통신 -> 서로 다른 네트워크로 통신이 가능

## 1번 방법

총 2개의 컨테이너를 띄울 것이다. 그리고 이 2개는 같은 네트워크에 속해 있다.

---
    # my-nginx-1 컨테이너 띄우기
    docker run --rm -d --name my-nginx-1 nginx

    # my-nginx-1 컨테이너 IP 확인 : 172.17.0.2 일 확률이 높음
    docker inspect my-nginx-1 | grep '"IPAddress"'

    # nginx-2 컨테이너 띄우기
    docker run --rm -it --name nginx-2 nginx bash

    # 배포판 확인
    cat /etc/issue

    # wget 설치
    apt update
    apt install -y wget

    # 2번 컨테이너에 접속하여 1번 컨테이너에 접속
    wget http://172.17.0.2/index.html

    # 확인
    cat index.html

my-nginx-1를 띄웠을때 nginx-2를 띄웠을때 각각 서로 다른 ip를 가진다. 1번은 0.2를 가지고 2번은 0.3을 가진다고 하자. 
<br>
여기서 2번 컨테이너 내부로 접근을 해서 wget을 이용해 172.17.0.2로 접근했다. 이것이 가능한 이유는 1,2번 컨테이너 모두 같은 네트워크에 속해있기 때문이다. 만약, 다른 네트워크 였다면 ip를 통한 접근이 불가능하다.

## 2번 방법

1번 방법은 도커의 기본 네트워크인 bridge에 소속되어 있다면 서로의 고유 IP(컨테이너)를 통해 통신할 수 있다.
<br>
하지만, IP가 변경되는 환경에서 사용하기에는 권장되지 않는다.

> ### 도커 기본 네트워크 bridge 와 사용자 정의 네트워크

기본 네트워크(bridge)는 도커가 설치될 때 자동으로 생성되는 네트워크이다. 기본 네트워크의 게이트웨이는 172.17.0.1이다.
<br>

여기서 도커에서 기본적으로 제공하는 네트워크가 아닌 사용자가 정의해 네트워크를 생성할 수 있다. 예를 들어, common을 사용자가 직접 생성한 네트워크라고 할 수 있다. 
<br>

중요한 점은 **사용자 정의 네트워크를 이용해 1번 방법의 문제점(ip가 변화)을 해결**할 수 있다는 것이다.
사용자 정의 네트워크는 기본 네트워크와 달리 **컨테이너 간에 DNS 이름 해석**(이름으로 컨테이너 식별이 가능)이 가능합니다
<br>

**즉, IP 주소 뿐만 아니라 컨테이너 이름으로도 통신이 가능하다는 것**이고 보안과 격리를 위해서는 기본 네트워크 대신 사용자 정의 네트워크를 사용하는 것이 좋다.

> 해당 스크립트로 docker의 기본 네트워크 bridge와 사용자 정의 네트워크를 확인,생성할 수 있다. 

    docker network ls
    docker network create common # 이름이 `common` 인 네트워크 생성
    docker network ls

    docker network inspect bridge | less # 디폴트 네트워크의 bridge 상세정보 보기, 게이트웨이가 172.17.0.1 임을 확인
    docker network inspect common | less # 사용자 정의 네트워크 common의 상세정보 보기, 게이트웨이가 172.18.0.1 임을 확인

> **Embedded DNS Server**

도커는 사용자 정의 네트워크를 생성할 때 내부에 내장 DNS 서버(127.0.0.11)를 자동으로 띄운다. 컨테이너가 http://nginx-1이라고 요청을 보내면, 이 내장 DNS가 nginx-1의 현재 IP가 무엇인지 찾아주는 원리이다.

> **컨테이너 이름으로 소통하기**

    docker network create common # 이름이 `common` 인 네트워크 생성

    # my-nginx-1 컨테이너 띄우기
    docker run --rm -d --name nginx-1 --network common nginx

    # nginx-2 컨테이너 띄우기
    docker run --rm -it --name nginx-2 --network common nginx bash

    # 배포판 확인
    cat /etc/issue

    # wget 설치
    apt update
    apt install -y wget

    # 2번 컨테이너에 접속하여 1번 컨테이너에 접속
    wget http://nginx-1/index.html

    # 확인
    cat index.html

1,2컨테이너를 띄우는데 기존의 birdge(172.17.0.1)로 띄워지지 않고 common(172.18.0.1)위에 띄워진다. 그리고 1,2번 컨테이너의 이름은 nginx-1(0.2)과 nginx-2(0.3)이다. 2번 컨테이너 내부로 접근해서 1번 방법과 다르게 1번 컨테이너의 ip주소로 접근하는 것이 아니라 이름으로 접근한다.

## 3번 방법

도커 컨테이너들은 호스트 OS의 포트(도커 호스트이다. 도커를 설치하면 생긴다. **어떤 네트워크에 있든 게이트웨이 IP(보통 .1)를 통해 호스트에 접근할 수 있다.**)를 통해서도 통신이 가능하다. -p 옵션으로 호스트 OS의 포트와 컨테이너의 포트를 연결할 수 있다.
<br>
예: -p 8080:80은 호스트의 8080 포트를 컨테이너의 80 포트와 연결합니다
<br>
컨테이너는 호스트 OS의 포트를 통해 다른 컨테이너와 통신할 수 있으므로, 컨테이너가 다른 네트워크에 있어도 통신이 가능한 장점이 있다.
---
    # my-nginx-1 컨테이너 띄우기
    docker run --rm -d -p 8080:80 --name my-nginx-1 nginx

    # nginx-2 컨테이너 띄우기
    docker run --rm -it --name nginx-2 nginx bash

    # 배포판 확인
    cat /etc/issue

    # wget 설치
    apt update
    apt install -y wget
    apt install -y iputils-ping

    # 게이트웨이를 통해서 도커 HOST에 접속가능한지 테스트
    ping 172.17.0.1

    # 2번 컨테이너에 접속하여 1번 컨테이너에 접속
    wget http://172.17.0.1:8080/index.html

    # 확인
    cat index.html

도커 host(172.17.0.1:8080)를 컨테이너 내부의 nginx1(80)을 맵핑한다. 현재 1,2 컨테이너는 다른 네트워크에 있다고 가정한다. 1,2번 컨테이너가 위의 1,2방법을 통해 소통할 수 없다. 
<br>
이때 2번 컨테이너에서 도커 host(172.17.0.1:8080)에 접근이 되는지 ping으로 확인해주고 접근이 가능하다는 것은 도커 host에 맵핑되어있는 1번 컨테이너로의 접근이 가능하다는 것이 된다. 
<br>
이런 방법을 통해 2번 컨테이너가 1번에 접근할 수 있다. 

> 3번 내용 작동원리 다시 정리 
- 포트 맵핑: nginx-1을 실행할 때 호스트의 특정 포트(8080)와 컨테이너 포트(80)를 연결. (-p 8080:80)
- 호스트 접속: nginx-2는 nginx-1의 IP를 몰라도, 호스트의 IP(게이트웨이 주소)와 맵핑된 포트로 요청을 보낸다.
- 전달: 호스트 OS가 받은 요청을 맵핑된 nginx-1 컨테이너로 전달하면서 2번과 1번 컨테이너가 소통한다.

> **host.docker.internal**

컨테이너에서 호스트의 IP를 알아내는 것은 생각보다 까다로울 수 있다. IP 주소 숫자를 외우지 않고 host.docker.internal이라는 이름으로 호스트에 접근할 수 있다. wget http://172.17.0.1:8080/index.html 이것을 wget http://host.docker.internal:8080/index.html 이렇게 바꿀 수 있다. 

>추가 지식, **만약 외부 인터넷에서 https://localhost:8080 요청이 들어올 경우**

1. host pc에 기본 8080이 있을 경우는 문제없다. 
2. host pc에 기본 8080이 없을 경우에는 접근할 수없다는 문제가 발생한다. 

3. docker host가 (8080:80)으로 존재할 경우

이때, 8080:80으로 맵핑이 되어있다고 가정하자. 외부에서 들어오는 8080 요청을 컨테이너 내부로 접근할 수 있게 하는 것이 가능하다. 
만약, 컨테이너가 같은 네트워크에 띄워진 상황이라면 고유 IP로 접근이 가능하지만 서로 다른 네트워크일 경우에는 **80 포트 컨테이너에서 172.17.0.1:8080으로 접근해 다른 컨테이너로 접근하는 것이 가능**하다.

즉, 도커 호스트에 맵핑이 되어있다면 서로 다른 네트워크에서 컨테이너끼리의 소통이 가능하다는 것이다.
