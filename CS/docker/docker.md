# Docker

기본 os에 프로그램을 설치하고 그 프로그램이 업데이트 되었을때, 새로운 프로그램이 설치되는 것이 아니라 기존 프로그램을 업데이트 시킨다. 여기서 생각을 이어나가면, docker를 왜 사용하게 되는지 알 수있다. 

> 예시
 
하나의 프로그램에서 A,B 2가지 요소가 필요하다. PC1에는 A,B가 존재해서 문제없이 프로그램을 실행할 수 있다. 하지만 PC2에서 똑같이 실행할려고 하니까 A는 있는데 B가 없는 문제가 발생했다. PC가 바뀌었다고 문제 없는 프로그램을 사용할 수 없게 된 것이다. 

<BR>
만약, PC2에 B를 설치해서 프로그램을 동작하는데 성공했다고 가정하자. 하지만, PC1에서 사용하는 B는 1.0버전이고 PC2에서 사용하는 버전은 2.0버전이다. 이렇게 버전차이가 날 때 프로그램 실행자체는 문제가 없을 수 있지만 만약 PC1과 PC2에서 사용하는 코드가 동일하다고 했을때 B 1.0과 B2.0에서 사용하는 라이브러리나 지원하는 기능의 차이가 발생했을때 문제가 발생할 수도 있다. 
<BR>
물론, 버전을 낮추어서 해결할 수도 있지만 2.0버전을 특정 OS가 지원하지 않을 경우도 생각해야 한다.
<BR>
<BR>
Docker는 서버의존성 문제도 해결하지만, 애플리케이션이 실행되는 환경(OS 설정, 라이브러리, 런타임, 패키지 버전 등)을 함께 묶어서 배포함으로써, 어디서 실행하더라도 동일한 환경을 보장하기 위해 사용한다.

## 서버 의존성 문제 해결

1. 서비스마다 서버를 따로 구성해서 의존성 문제를 해결하고자 했다. -> 하지만, 비용문제와 가성비 측면에서 너무 떨어진다는 문제점이 있었다. 
예를 들어, b서버를 구매해서 프로그램을 실행했는데 30%만 사용하고 70%는 놀고 있다면 굉장한 낭비가 된다. 70%를 다른 서비스에 부여하면 해결할 수도 있지만, 다른 의존성 문제 때문에 골치
2. 1번에 대한 해결책으로, 한 대의 pc에서 os를 나누고 하드웨어의 리소스를 분할해서 사용하는 방식(가상머신)을 사용했다. => 하지만, 각 VM마다 Guest OS 전체를 띄워야 하기 때문에 굉장히 무겁고 비효율적인 구조가 된다. 
<br>
예를 들어,
---
    Host OS
    └ Hypervisor
        ├ Guest OS 1
        │   └ App
        ├ Guest OS 2
        │   └ App

이런 구조이다. 각 vm 마다 커널,시스템 프로세스,OS 파일들 전부 따로 존재함.

3. 2번에 대한 해결책이, 커널을 공통되게 하나로 두고 여러 os를 사용하는 것이다.(docker)=> **하나의 OS 커널 위에서 격리된 프로세스를 실행하는 기술** 혹은, Docker는 Host OS의 커널을 공유하면서, 프로세스를 격리된 환경에서 실행하는 컨테이너 기술이다.
----
    Host OS
    └ Docker Engine
        ├ Container 1
        ├ Container 2



- 어떤 컨테이너 엔진을 사용해도 docker명령어를 사용하기에 docker명령어를 익혀두면 다른 컨테이너도 사용할 수 있다.

## Docker Imgaes vs container

터미널이라는 프로그램을 예시로 들어서 설명해보자면, 우리 컴퓨터(os)에서 터미널 프로그램을 여러개 띄우는 것이 가능하다. 하나의 프로그램을 실행가능하게 바꾸어 프로세스로서 동작하게 만든 것이다. 
**여러가지 프로세스가 있지만 사용된 프로그램은 단 1개이다.**

<br>
<br>

- 도커 이미지 = 프로그램, class같은 개념이다.
- 도커 컨테이너 = 프로세스 , class로 만들수 있는 여러 객체의 개념이다.

즉, **하나의 이미지에서 여러 컨테이너를 만들 수 있다는 것이다.**

- 이미지를 다운로드 하는 방법 -> docker pull을 사용
- 이미지를 어디서 다운 받는가? => 설정하지 않으면 기본적으로 docker hub(이미지 저장소)에서 사용한다. 
- 이미지는 누구나 만들어 올릴 수 있다.

## 주 프로세스

주 프로세스는 **컨테이너 내부에서 실행되는 대표 프로세스** 라고 생각하면 된다.
<br>
hello라는 컨테이너가 os위에 띄워져있고 해당 컨테이너의 기능은 hello를 출력하는 것이다. hello라는 컨테이너에서 주 프로세는 print(hello)라고 생각하면 된다. 물론, os적으로 print를 하기 위한 여러가지 프로세스들이 필요하겠지만 신경쓰지 않는다고 보면된다. 오직 print(hello)를 하기 위해서 hello컨테이너가 올라가고 만약, hello출력에 성공하면, 컨테이너는 종료된다.

> 흐름도

이미지 생성 -> 컨테이너 생성 -> 주프로세스 실행 -> 컨테이너 종료

- docker ps = 실행중인 컨테이너 확인
- docker ps -a = 실행했던 컨테이너 목록 확인

## Port(포트)

포트는 프로그램 수준에서 네트워크를 식별하는 것이다.(식별자의 역할)
<br>
예를 들어, pc는 nic에서 ip를 통해 외부 인터넷과 소통한다. pc 내부에는 a,b,c 프로그램이 실행되기 위해서는 인터넷이 필요하다. nic으로 인터넷과 연결되었다고 해도 a라는 프로그램에 인터넷이 연결되기 위한 식별자가 필요하다. 그것이 포트다.

### 만약, pc 내부에 컨테이너가 있을 경우 포트는 어떻게 설정되는걸까?

pc 내부 a프로그램 포트는 8080이다. 컨테이너 내부의 a프로그램도 8080을 가진다. 인터넷을 연결해서 8080을 찾는다고 하면 컨테이너 내부에 있는 a프로그램을 찾을 수 있을까? 없다. 만약 pc의 a프로그램이 없어지더라도 컨테이너 내부의 a프로그램을 찾을 수 없다. host OS와 격리되어있기 때문이다. 

<br>
<br>
그렇다면, 컨테이너 포트를 노출시키는 방법이 필요하고 외부로 들어난 포트와 컨테이너 내부 프로그램의 포트를 서로 맵핑을 해줘야 한다.

예를 들어, docker ps -a 를 통해 연결된 것을 확인 할 수 있다. .0.0.0.0 8080-> 80/tcp 이런식으로 만들어진다. 
<br>
여기서 https://localhost:8080을 하면 컨테이너랑 이어지게 되는 것이고 컨테이너의 8080은 내부 프로그램 80포트와 연결이 되게 되는 것이다.

----

    docker run -p 8080:80 nginx

    내 PC의 8080 요청을 컨테이너 내부의 80 포트로 전달

## 컨테이너 네트워크

컨테이너는 독립 IP,독립 포트 공간,독립 네트워크 namespace를 가진다, 그래서 컨테이너 a도 80을 가지고 b도 80을 가질 수 있다. 

- 도커 네트워크가 동일하게 존재해서 컨테이너끼리 통신이 가능하다.
