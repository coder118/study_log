# AWS 간단 설명

- AWS(Amazon Web Services)는 아마존에서 제공하는 클라우드 컴퓨팅 서비스이다. 서버, 스토리지, 데이터베이스, 네트워킹, 분석, 기계 학습, 인공 지능, 모바일, 개발 도구 등 다양한 서비스를 제공한다.
전 세계 여러 리전(region)에 데이터 센터를 두고 있어 글로벌 서비스 제공이 용이하며 높은 확장성과 안정성을 제공한다.

# terraform 


- HashiCorp에서 개발한 인프라스트럭처를 코드로 관리하는(IaC, Infrastructure as Code) 오픈소스 도구이다. 선언적 구성 파일(aws cli는 명령형이다)을 사용하여 클라우드 리소스를 생성, 관리, 업데이트할 수 있다.
    - 명령형인 cli보다 사용하기 편하다.


- 마치 HTML 태그로 엘리먼트를 만들듯이 서버 자원을 직관적으로 다룰 수 있으며, AWS, Azure, Google Cloud 등 다양한 클라우드 제공업체를 지원한다.
    - 하나의 도구로 여러 클라우드를 통제할 수 있다.

- 버전 관리가 가능하여 인프라 변경 사항을 추적하고 협업 가능하다. 코드로 인프라를 관리하므로 반복 가능하고 일관된 환경을 구축할 수 있습니다.
    - 관리하기 편리하다.

- 리소스 간의 종속성을 자동으로 처리하여 올바른 순서로 리소스를 생성한다. 의존성에 따라서 VPC를 먼저 만들고 그 이후에 EC2를 만드는 식으로 작동합니다.
**예를 들어,** AWS 콘솔을 통해 EC2 인스턴스 100개를 생성하는 것은 반복적이고 시간이 많이 소요되는 작업이다. 테라폼을 사용하면 코드로 인프라를 정의하여 단 몇 줄의 코드로 100개의 EC2 인스턴스를 생성한다.


# AWS 대표 서비스

- IAM : 사용자 계정 및 권한 관리 서비스
- EC2 : 가상 서버 인스턴스 제공 서비스
- S3 : 확장성 있는 객체 스토리지 서비스
- RDS : 관계형 데이터베이스 서비스
- Lambda : 서버리스 컴퓨팅 서비스
- DynamoDB : NoSQL 데이터베이스 서비스
- CloudFront : 콘텐츠 전송 네트워크(CDN) 서비스
- VPC : 가상 사설 클라우드 네트워크 서비스
- Route 53 : DNS 웹 서비스
- CloudWatch : 모니터링 및 관찰 서비스

## 루트 계정과 IAM 계정의 관계

- 루트 계정 :AWS 계정을 처음 생성할 때 만들어지는 최상위 관리자 계정으로, 모든 AWS 서비스와 리소스에 대한 완전한 접근 권한을 가지고 있다.

루트 계정을 사용하면 편하겠지만,**보안상의 이유로 루트계정은 일상적인 작업에 사용하지 않는 것이 권장**된다. 

- IAM 계정 : 루트계정에서 생성한 사용자 계정(루트 계정 하나로 여러개 만들 수 있음)으로, 특정 권한만 부여받을 수 있다.

IAM을 통해 여러 사용자에게 필요한 권한만 부여하여 보안을 강화할 수 있다.
<br>
예를 들어, 일반적으로 관리자 권한을 가진 IAM 사용자를 생성하여 대부분의 관리 작업을 수행한다. 이때, 루트계정은 IAM 사용자 생성, 계정 해지, 결제 정보 변경 등 중요한 작업에만 사용하는 것이 좋다.

그럼, 루트로 하면 편한데 굳이 IAM에 관리자 권한을 왜 줘야하는데? 라는 질문

## 루트계정이 있는데 admin IAM 계정을 만들어서 사용하는 이유


1. 루트계정은 AWS 계정의 모든 권한을 가진 최상위 계정으로, 해킹 시 심각한 보안 위험이 있다. 
2. admin IAM 계정은 AdministratorAccess 정책을 통해 대부분의 권한을 가지지만, 일부 중요 작업(계정 삭제, 결제 방법 변경 등)은 제한된다. IAM 계정은 필요시 권한 제한이나 삭제가 가능합니다.
또한, 루트계정 활동은 CloudTrail에서 감사하기 어렵지만, IAM 계정의 모든 활동은 추적 및 감사가 가능하다.
3. 별개의 이유로, 여러 관리자가 필요한 경우 각각에게 별도의 IAM 계정을 부여하여 책임 소재를 명확히 할 수 있다.
4. 가장 정석적인 AWS 보안 모범 사례에서는 루트계정은 MFA를 설정하고 일상적인 작업에는 사용하지 않도록 권장한다.


# 정책과 권한

## 정책은 권한의 묶음.

- 권한1+ 권한 2 = 정책(json) => 사용자, 그룹, 역할에 연결

AWS IAM에서 정책(Policy)은 하나 이상의 권한을 모아놓은 집합이다.
정책은 JSON 형식으로 작성되며, 사용자, 그룹 또는 역할에 연결하여 권한을 부여한다.
AWS는 일반적인 사용 사례에 대한 다양한 관리형 정책을 제공한다. 

> 예를 들어,

- 관리형 정책 : AdministratorAccess, ReadOnlyAccess, AmazonS3FullAccess 등 AWS에서 관리하며 자동으로 업데이트됩니다.
- 사용자 지정 정책 : 직접 생성하여 특정 요구사항에 맞는 권한을 정의할 수도 있습니다.

**정책은 "누가", "무엇을", "어디서", "어떤 조건에서" 수행할 수 있는지 정의**한다.
최소 권한 원칙에 따라 필요한 권한만 부여하는 것이 보안 모범 사례입니다.

## 권한을 사용할 수 있는 대상

- AWS IAM에서 권한을 부여할 수 있는 대상은 크게 사용자(User), 그룹(Group), 역할(Role)이 있다.

- 사용자1 + 사용자2 = 그룹 1
- 사용자1 + 사용자3 = 그룹 2

1. 사용자(User): 개별 AWS 계정으로, 특정 개인이나 애플리케이션에 해당 / 
고유한 자격 증명(사용자 이름과 비밀번호)을 가지며 콘솔 로그인이 가능 / 장기 자격 증명(액세스 키)을 생성하여 API 호출에 사용할 수 있다.

2. 그룹(Group): 여러 사용자를 모아놓은 집합 / 그룹에 정책을 연결하면 그룹의 모든 사용자에게 해당 권한이 부여 / 사용자 **권한 관리를 단순화하고 일관성을 유지**하는 데 도움이 됩니다. **한 사용자는 여러 그룹에 속할 수 있습니다.**


3. 역할(Role): 특정 AWS 리소스나 **외부 사용자가 임시로 사용**할 수 있는 권한 세트 / EC2 인스턴스, Lambda 함수 등의 **AWS 서비스에 권한을 부여**할 때 사용 / 
페더레이션 사용자(외부 ID 제공자를 통해 인증된 사용자)에게 권한을 부여할 때 사용한다 / 교차 계정 액세스를 관리할 때 사용한다.

4. 리소스 기반 정책: 일부 AWS 서비스(S3, SNS, SQS 등)는 리소스 자체에 정책을 연결할 수 있다./ 이를 통해 해당 리소스에 대한 액세스를 직접 제어할 수 있습니다.

## EC2가 권한을 가지는 방법

- EC2 인스턴스 : IAM 역할(Role)을 통해 AWS 서비스에 대한 권한을 가질 수 있다.
    - 역할을 연결하면 해당 인스턴스에서 실행되는 애플리케이션이 **액세스 키 없이도 AWS API를 호출할 수 있다**.
    <br>
    => 이 방식은 액세스 키를 EC2 인스턴스 내에 저장하는 것보다 훨씬 안전.

- EC2 인스턴스 프로필 : 역할이 EC2 인스턴스에 연결할 수 있다. 인스턴스 생성 시 역할을 지정하거나 실행 중인 인스턴스에 역할을 나중에 연결할 수 있다.

# IAM 리전(region)

IAM 서비스의 주소가 us-east-1로 시작하는 이유는 IAM이 글로벌 서비스이기 때문이다.
IAM은 특정 리전에 귀속되지 않는 글로벌 서비스로, AWS의 모든 리전에서 동일하게 적용된다.
<br>
하지만 기술적으로는 us-east-1(버지니아) 리전에서 관리되며, 이 리전은 AWS의 첫 번째이자 가장 큰 리전이다. IAM 설정(사용자, 그룹, 역할, 정책 등)은 모든 리전에서 동일하게 적용됩니다.
<br>

반면, EC2, S3, RDS 등 **대부분의 AWS 서비스는 특정 리전에 귀속되어 있어** 리전마다 독립적으로 관리된다. 글로벌 서비스의 다른 예로는 Route 53(DNS 서비스), CloudFront(CDN), WAF(웹 애플리케이션 방화벽) 등이 있다.


# VPC

가용영역 1,2가 있다고 했을때 1과 2가 소통을 하고 싶어도 네트워크가 다르면 소통할 수 없다. 이를 해결하기 위해 사용하는 것이 vpn이다. 1번이 vpn으로 연결하고 2번도 마찬가지이다. 그렇게 되면, 1,2는 같은 네트워크 안에 있는 것이 되면서 서로 소통하는 것이 가능하다.
<br>
vpc는 리전에 소속된다.

## 서브넷

vpc 내부를 또 독립적인 공간으로 쪼개는 것이 가능하다. vpc 아래 하위 네트워크이다.

> 10.0.0.0/16 
네트워크에서 관리하는 IP 개수 : 255 * 255
대역 : 10.0.0.0 ~ 10.0.255.255

> 10.0.0.0/24
네트워크에서 관리하는 IP 개수 : 255
대역 : 10.0.0.0 ~ 10.0.0.255

<br>
위와 같은 방식으로 쪼개는 것이 가능하다. 16의 경우 쪼갤 수 있는 ip개수가 255*255이 것이다. 많은 양의 ip가 필요하면 16을 사용하면 되는 것이고 아니라면 24를 사용하는 것

## 라우트 테이블 

1. 독립적인 이정표: 내가 만든 VPC(vpc_1)와 다른 사람이 만든 VPC는 당연히 서로 다른 네트워크 주소를 쓰고, 다른 길로 데이터를 보내야 한다. 따라서 라우트 테이블은 특정 VPC에 종속되어 그 내부의 길을 안내해 주는 역할

2. 서브넷과의 연결: 라우트 테이블은 VPC 안에 쪼개져 있는 방들(서브넷들)에 직접 깃발처럼 꽂아서 사용
=> 라우트 테이블이 서브넷을 가리키는 것이 아니라, 서브넷이 라우트 테이블을 바라보고(참조하고) 있는 것이다.

<br>
예를 들어, 
- 서브넷이 라우트 테이블을 가리킨다는 말: 서브넷 안에 있는 서버(EC2 등)들이 "나 저기 바깥 인터넷 세상(0.0.0.0/0)으로 가고 싶은데 어디로 가야 하지?" 할 때, 자신에게 연결된 라우트 테이블(이정표)을 확인.

- 라우트 테이블은 '목적지(IP)'를 가리킨다: 라우트 테이블 안에는 서브넷 이름이 적혀 있는 게 아니라, "IP 주소가 0.0.0.0/0(모든 외부 인터넷)인 곳으로 가려면 igw_1(인터넷 게이트웨이 문)으로 가라"는 내비게이션 안내만 적혀 있다.
