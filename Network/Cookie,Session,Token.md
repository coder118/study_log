# Cookie

Key-value 형식의 문자열로 구성되어있다.

클라이언트가 특정 웹 사이트를 방문했을 경우, 해당 사이트의 서버를 통해 클라이언트의 브라우저에 설치되는 작은 기록 정보 파일이다.

각 사용자마다의 브라우저에 정보를 저장함으로 고유 정보 식별이 가능.
=> 누가 누군지 알수 있음

> 쿠키 관련된 헤더 2가지

1. Set- Cookie 헤더

Response에 사용되며 서버에서 클라이언트로 쿠키 전달.

=> 쿠키 만들어서 보내는 곳

2. Cookie 헤더

Request에서 사용되며 클라이언트가 서버에서 받은 쿠키를 저장한다.
request시에 서버로 전달한다.

=> 쿠키를 받아서 가지고 있다가 다시 또 서버에 주는 곳


### 쿠키의 인증 방식

![쿠키 인증 과정](/Network/network_images/img1.daumcdn.png)

1. 브라우저(클라이언트)가 서버에 request함
2. 서버는 requset에 대한 response를 작성한다. 그리고 클라이언트 측에 저장하고 싶은 정보를 응답 헤더의 Set-Cookie에 담는다.
3. 이후 클라이언트는 request할때마다 저장된 쿠키를 request헤더에 담아 보낸다.
서버는 쿠키에 담긴 정보를 바탕으로 해당 요청의 클라이언트가 누군지 식별하거나 정보를 바탕으로 추천광고를 띄움.

=> accept cookie all 그거 같은데 외국 웹 사이트에 들어가면 나오는거.

### Cookie방식의 단점

- 보안에 취약하다. (request시에 쿠키의 값을 그대로 보내기때문에 개인정보 같은 민감한 정보가 있다면 유출,조작 당할 수 있다.)
-  쿠키에 용량제한이 있다
- 웹 브라우저마다 쿠키에 대한 지원형태가 달라서 브라우저간 공유가 불가능
- 쿠키 사이즈가 커질 수록 네트워크 부하 심해진다.



# Session

위 쿠키의 보안적 이슈때문에, 세션은 비밀번호 등 클라이언트의 민감한 정보를 브라우저가 아닌 **서버 측에 저장하고 관리**한다.

서버 메모리, 서버의 로컬 파일, DB에 저장하기도 한다.

결론은, **민감한 정보는 클라이언트로 보내지 않고 서버에서 관리**.


> 세션 객체는 어떤 형태로 이루어져 있을까?

세션 객체는 {  Key에 해당하는 SESSION ID + 대응하는 Value  } 로 구성되어 있다.
<br>
Value에는 세션 생성 시간, 마지막 접근 시간 및 User가 저장한 속성 등 이 Map 형태로 저장된다


### Session 인증 방식

![세션 인증 방식](/Network/network_images/img1.daumcdn-1.png)

1. 유저가 로그인하면 session이 서버 메모리(or DB)에 저장. 이때 Session Id를 기준으로 세션을 식별한다.
2. 서버에서 브라우저 쿠키에다가 Session Id를 저장
3. 쿠키에 정보가 담겨있다. 즉, request할때 Session Id를 쿠키에 담아 서버로 보낸다.
4. 클라인트가 보낸 Session Id와 서버 메모리로 관리하는 Session Id를 비교하며 인증한다.

### Session 방식의 단점

- Session Id가 쿠키에 담겨서 노출된다고 해도 의미 없다. 하지만 Session ID 자체를 탈취해서 클라이언트인척 위장할 수 있다는 한계가 있다.(서버에서 IP특정을 통해 해결 가능)
- 서버 저장소를 사용하는 요청이 많아지면 서버에 부하가 심해진다.


# Token

> 기존 서버(세션) 기반 인증 시스템

위의 session처럼 서버에서 사용자 인증정보 관리하는 것이다. 그러다 보니, 클라이언트로부터 **요청을 받으면 클라이언트 상태를 계속 유지해놓고 사용(Stateful)**

이는, 사용자가 증가함에 따라 성능 문제를 일으킬 수 있으며 확장성이 어렵다.

> 토큰 기반 인증 시스템

session기반의 단점을 극복하기 위한 시스템.
인증 받은 사용자에게 토큰을 발급하고, 로그인이 필요할때 헤더에 토큰을 함께 보내(requset할때를 말함) 인증받은 사용자인지 확인.

이는 **상태를 유지하지 않으므로 Stateless한 특징**
------
token은 앱과 서버가 통신 및 인증할때 많이 사용된다. 웹에는 쿠키와 세션이있지만 앱에서는 없기 때문이다.

### Token 인증 방식


![토큰 인증 방식](/Network/network_images/img1.daumcdn-2.png)

1. 사용자가 로그인 한다.(request)
2. 서버측에서 사용자(클라이언트)에게 유일한 토큰을 발급한다.(response)
3. 클라이언트는 서버에서 받은 token을 쿠키, 스토리지에 저장해두고, 서버에 request할때마다 해당 token을 HTTP요청 헤더에 포함시켜 전달한다.
4. 서버는 전달받은 토큰을 검증하고 요청에 응답한다. 토큰에는 요청한 사람의 정보가 담겨있기에 서버는 DB를 조회하지 않고 누가 요청하는지 알 수 있다.

### Token방식의 단점

- 쿠키/세션과 다르게 데이터가 길어서 네트워크 부하가 심해질 수 있다.
- payload 자체는 암호화되지 않아서 유저의 중요한 정보를 담을 수 없다.
- token을 탈취당하면 대처가 어렵다.(그래서 사용기간을 제한하는 방식)
